# 클라이언트 - 서버
클라이언트 - 외뢰인으로, 서비스를 요청하는 쪽
- 요청을 보내는 쪽으로, 요청은 N개가 존재한다.
- 요청을 보내기만 하는 과정이라 리소스 소모가 적다
- 요청을 시작점 + 헤더 + 바디의 과정으로 요청이 간다

서버 - 제공자로, 요청에 응답하는 쪽
서버는 클라이언트가 될수 없다? : x  = 서버가 다른서버에 요청을 보내면 서버+클라이언트 상태가 동시에 있을수 있다.
- 응답은 요청에 대한 분석과, 이에 대한 응답이 수반됨으로 리소스 소모가 많다.
- 응답은 1개 혹은, 몇개가 존재한다.
- 응답 역시 헤더+ 바디를 포함한다.

클라이언트 - 서버 모델은 요청과 응답의 반복으로 이루어진 모델
잠재적 단일 장애 지점(SPOF) - 한부분의 실패가 전체 시스템의 중단을 초래한 요소(웹 서버 다운시, 클라이언트, DB서버 모두 이용불가)
클라이언트가 다수 일시를 대비하여 다중대처구조를 구현하여(3-Tier 계층구조) 유연하게 대처할수 있어야 한다.

"웹 브라우저에서 특정 주소를 치면 어떤식으로 서버를 응답하는가? = DNS가 어떤식으로 작동하는가?"

# Http Protocal

- Protocal : 네트워트 통신상 상호작을 위해 서로 지켜야할 약속과 규칙

# Rest API 특징
REST는 Representational State Transfer(표현 상태 전송)의 약자로, 
웹 서비스에서 자원(Resource)을 이름(URI)으로 식별하고 해당 자원의 상태 정보를 HTTP 프로토콜을 통해
주고받는 아키텍처 스타일(규칙)이며, API는 Application Programming Interface(애플리케이션 프로그래밍 인터페이스)의 
약자로, 소프트웨어 구성 요소들이 서로 통신할 수 있게 하는
인터페이스(규약)입니다. 따라서 REST API는 REST 아키텍처 스타일을 따르는 API로, 
HTTP 요청을 통해 자원(데이터)을 조회, 생성, 수정, 삭제하는 방식이며, 클라이언트가 서버와 통신하는 가장 보편적인
방법입니다.

REST란? (Representational State Transfer)
자원(Resource): 웹 상의 모든 데이터(문서, 이미지, 사용자 정보 등)를 자원이라 칭하며, 
이는 URI로 표현됩니다 (예: /users/123).
상태(State): 자원의 현재 상태(정보)를 의미하며, 이 상태를 클라이언트와 서버가 주고받습니다.
전송(Transfer): 클라이언트가 서버에 자원을 요청하면, 
서버는 해당 자원의 상태를 표현(Representation)하여 응답합니다 (주로 JSON, XML 형태).
핵심 원칙: URI로 자원을 식별하고, HTTP 메소드(GET, POST, PUT, DELETE)를 사용하여 자원에 대한 행위(CRUD)를 표현합니다.

API란? (Application Programming Interface)
소프트웨어 간의 상호작용을 위한 규칙과 방법의 집합입니다.
프론트엔드 개발자가 백엔드 서버의 데이터에 접근하거나, 다른 서비스의 기능을 이용할 수 있게 해주는 문지기 역할을 합니다.

REST API의 특징
클라이언트-서버 구조: 자원이 있는 쪽(서버)과 요청하는 쪽(클라이언트)이 분리됩니다.
무상태(Stateless): 서버는 클라이언트의 이전 요청에 대한 상태 정보를 유지하지 않습니다. 각 요청은 독립적입니다.
캐시 가능(Cacheable): 응답을 캐싱하여 성능을 향상시킬 수 있습니다.
계층화(Layered System): 중간 서버(프록시, 로드밸런서 등)를 추가할 수 있습니다.
URI로 자원 명시: /users, /products/1과 같이 명확한 URI로 자원을 표현합니다
